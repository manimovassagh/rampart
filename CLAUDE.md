## Project Overview

**Rampart** is a lightweight, modern identity and access management (IAM) server — an open-source alternative to Keycloak with better UX and a modern stack.

- **Repo:** https://github.com/manimovassagh/rampart
- **License:** TBD (leaning AGPL for open-core protection)
- **Goal:** Open-source core with future enterprise/commercial tier (Red Hat model)

## Tech Stack

| Layer | Tech | Rationale |
|-------|------|-----------|
| Backend | Go | Single binary, low memory, fast startup |
| Admin UI | React + Vite + Tailwind | Modern SPA, no Vercel dependency, embeds in Go binary |
| Login/Consent UI | Go templates + Tailwind | Server-rendered by Go, themeable per-tenant, no JS framework needed |
| Database | PostgreSQL | Battle-tested, JSONB for flexibility |
| Cache/Sessions | Redis/Valkey | Session mgmt, token blacklisting |
| Config | YAML + API-first | GitOps-friendly |

## Architecture Decisions

- Single binary deployment (UI embedded in Go binary)
- Plugin system: TBD (WASM vs gRPC vs Go-only)
- Event sourcing for audit trail
- Multi-tenant by default (organizations/realms)

## Competitive Positioning

"The lightweight, modern Keycloak — single binary, beautiful UX, enterprise-ready."

Key differentiators vs competitors:
- vs Keycloak: lightweight, modern UI, easy to extend
- vs Ory: batteries-included, not headless
- vs Zitadel: PostgreSQL (not CockroachDB), extensible plugin system
- vs Authentik: Go performance (10x throughput), single binary

## Open Core Split

### Community (Free)
- Full OIDC/OAuth 2.0/SAML provider
- User management, groups, roles, admin dashboard
- MFA (TOTP, WebAuthn), social login
- REST API, webhook events, basic audit log
- CLI tool, themeable login pages

### Enterprise (Paid — future)
- HA clustering, SCIM 2.0 provisioning
- Advanced MFA policies (adaptive, risk-based)
- Compliance dashboards (SOC2, GDPR, HIPAA)
- Management plane for multi-cluster
- LDAP/AD bridge, priority support + SLA

## Development Rules

- All plans, decisions, and history stay in this repo (never outside project root)
- No code without tests for core auth flows
- Keep the binary small — avoid unnecessary dependencies
- API-first: every feature must be accessible via REST API before building UI
- **Keycloak-level quality** — this is not a toy project. Every feature must be production-grade, following RFCs and security best practices at the level of Keycloak, Ory, and Zitadel.

### Small Moves, Verify, Then Proceed
- **Always work in small incremental steps.** Do one thing, verify it works (compile, test, lint), then move to the next.
- **Check with available tools** (MCP servers, docs, web search) before assuming anything — verify APIs, library usage, config formats, etc.
- **Never go deep into a rabbit hole.** If something isn't working after a reasonable attempt, stop, reassess, and ask the user.
- Each step should be independently correct before building on top of it.

### Git & Commit Rules
- **Commit naturally like a human developer.** Small, logical commits with clear messages.
- Commit messages should read like a real developer wrote them — no AI fluff or over-explanation.
- **Do NOT add Co-Authored-By, "generated by AI", or any AI attribution to commits.** The repo should look like a normal human-driven open-source project.
- Do NOT add Claude or any AI assistant as a contributor, co-author, or collaborator on the repo.
- Commit when a logical unit of work is done, not after every tiny change.
- Use conventional-ish messages: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`, `chore:` prefixes.

### Branching Strategy
- **NEVER push directly to main. No exceptions.**
- `main` is always stable and deployable — treat it as production.
- **All work happens on feature branches.** Create a branch before writing any code.
- Branch naming: `feat/short-description`, `fix/short-description`, `chore/short-description`.
- Every branch merges to main via a PR — no direct commits to main, ever.
- PRs need a clear title and description of what changed and why.
- **All PRs must be reviewed and approved by the project owner before merging.** No self-merging. The owner needs to understand and approve every code change.
- Delete branches after merge — keep the repo clean.
- Keep branches short-lived — don't let them drift far from main.
- Rebase on main before merging if the branch is behind.
- If main needs a hotfix, still use a branch: `fix/urgent-description` → PR → merge.

### Issue-Driven Development
- **Every feature, bug fix, or task must have a GitHub issue before work starts.**
- Issues are the single source of truth for what's planned, in progress, and done.
- Branch names should reference the issue: `feat/42-oidc-auth-code-flow`.
- PRs must reference the issue they close: `Closes #42`.
- Use labels to categorize: `feature`, `bug`, `docs`, `security`, `infra`, `phase-1`, `phase-2`, `phase-3`.
- Use milestones to group issues by release phase.

### Security-First Development
- This is an IAM product — **security is not optional, it's the product.**
- Never store secrets, passwords, or tokens in plaintext — not in code, config, or logs.
- All crypto must use well-known, audited libraries — never roll custom crypto.
- Validate and sanitize ALL input at system boundaries.
- Default to deny — whitelist, don't blacklist.
- Every auth flow must have tests covering both happy path and attack vectors.
- Log security-relevant events (login, failed auth, permission changes) but never log sensitive data.

### Code Quality
- Follow idiomatic Go — use `gofmt`, `go vet`, `golangci-lint`.
- No `interface{}` / `any` unless absolutely necessary — use strong types.
- Error handling: wrap errors with context (`fmt.Errorf("doing X: %w", err)`), never swallow errors silently.
- Keep functions short and focused. If a function needs a comment explaining what it does, it should probably be split.
- No dead code, no commented-out code in main branch.

### Dependency Management
- **Minimize dependencies.** Every dependency is a supply chain risk — especially for an IAM product.
- Before adding a dep: check if the stdlib can do it, check maintenance status, check license compatibility.
- Pin exact versions. Run `go mod tidy` regularly.
- Audit deps periodically for known vulnerabilities.

### Testing Strategy
- **Unit tests** for all business logic (domain layer).
- **Integration tests** for database operations and auth flows.
- **End-to-end tests** for critical paths (login, token issuance, token refresh).
- Tests must be runnable with a single command (`go test ./...`).
- Use table-driven tests (idiomatic Go).
- Test names should describe the scenario: `TestLoginFlow_InvalidPassword_ReturnsUnauthorized`.

### Documentation
- Code should be self-documenting. Add comments only when the "why" isn't obvious.
- Public APIs (exported Go functions, REST endpoints) need godoc/OpenAPI docs.
- Architecture decisions go in CLAUDE.md (this file), not scattered markdown files.
- README.md is for users — keep it clean, practical, and up to date.

### File & Package Organization
- **Go code lives in the project root.** `main.go` at root, packages in `internal/`.
- **Frontend (React + Vite) lives in `client/`** — added later when UI work starts.
- `internal/` for private packages — keeps the public API surface small.
- One concern per package. Don't put user management and OIDC in the same package.
- Avoid circular dependencies — if two packages need each other, extract a shared interface.

### CI/CD & Quality Gates
- PRs must pass ALL checks before merge: lint, test, security scan, build.
- Use modular/reusable GitHub Actions workflows — not one giant YAML.
- Pin Actions by SHA, not tag (supply chain protection).
- Every PR runs: `golangci-lint`, `go vet`, `go test -race`, `govulncheck`, `gosec`.
- Test coverage must not drop below the established threshold.
- Branch protection on `main`: require PR reviews, require status checks, no force push.
- CODEOWNERS file to enforce review requirements.
- On release tags: cross-compile, create GitHub release, push Docker image.

### When Stuck
- Don't guess — look up official docs, RFCs (OAuth 2.0 is RFC 6749, OIDC is built on top).
- Check how Ory, Zitadel, or other mature projects solved the same problem.
- If blocked for more than 2 attempts, stop and discuss with the user.
- Prefer boring, proven solutions over clever ones.
